❌ PROBLEM 2 — CLEAR NAME

Tracked factor counts reset to zero after sign-out and sign-in.

Observed behavior:

You increase factor counts (e.g. Physical activity = 2, Mindfulness = 2).

You sign out.

You sign back into the same account.

Factor counts reset to 0.

Factors themselves still exist, but their daily values are lost.

This confirms:

❌ Factor values are not persisted per day

✅ Factor definitions are persisted

🧠 What is ACTUALLY wrong (root cause)

This is not the same bug as the check-ins, but it’s related.

What’s likely happening:

Factors are stored as static definitions (name, icon, type)

Factor counts are stored in temporary UI state

On sign-in or reload:

The app reloads factor definitions

But does NOT reload factor values for the selected date

So the UI defaults them to 0

In short:

You persist the factor list, but not the factor entries per day.

✅ What the CORRECT behavior must be

For each factor count:

It must be stored per user + per date

It must survive:

Reload

Navigation

Sign-out

Sign-in

When the user returns to a date:

The saved count must rehydrate into the UI

This is the same pattern that daily check-ins should follow.

✅ REQUIRED DATA MODEL (conceptual, simple)

Each factor entry needs at least:

userId (or anonymousId)

date (YYYY-MM-DD)

factorId

count (number)

Without this, persistence will always break.

Goal:
Tracked factor counts must persist reliably across sign-out and sign-in, just like other user data.
